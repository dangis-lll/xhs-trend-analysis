# 小红书趋势分析功能设计方案

## 1. 目标

构建一个面向运营调研的小红书趋势分析工具。工具每天定期围绕某个领域的一组关键词抓取搜索结果页数据，经过清洗、降重、OCR/多模态解析和大模型分析后，输出可量化、有证据、可复盘的趋势报告。

第一阶段只依赖搜索结果页可直接获取的信息，不默认进入详情页，以降低账号风控风险。

核心问题：

- 这个领域最近哪些话题正在升温？
- 哪些标题、封面、场景、痛点、产品词更容易获得互动？
- 哪些帖子值得作为案例拆解？
- 哪些内容方向适合后续选题、投放或创作？

## 2. 数据来源

当前脚本可稳定获取的字段：

- `keyword`：搜索关键词
- `rank`：搜索结果页解析顺序
- `title`：笔记标题
- `author`：发布人
- `publish_time`：页面原始发布时间，例如 `3天前`、`6-18`、`2025-12-03`
- `publish_date`：标准化日期，格式 `YYYY-MM-DD`
- `link`：笔记链接
- `note_id`：笔记 ID
- `cover_url`：封面图链接
- `like_count`：搜索结果页展示的点赞数
- `visible_text`：卡片可见文本
- `data_attrs`：卡片 DOM 上可见的 `data-*` 属性

不默认采集的字段：

- 评论内容
- 正文全文
- 收藏数
- 分享数
- 作者主页数据
- 精确发布时间

这些通常需要进入详情页或调用接口，稳定性低，且更容易触发风控。

## 3. 每日运行流程

建议采用每日固定时间运行，例如每天上午 9 点和晚上 8 点各一次。如果只做轻量调研，每天一次即可。

流程：

1. 生成当天关键词列表。
2. 对每个关键词运行搜索页采集脚本。
3. 保存原始采集结果，禁止覆盖历史数据。
4. 下载封面图。
5. 对封面图做 OCR 或多模态解析。
6. 合并文本、图片、互动和时间字段。
7. 做规则降重和语义降重。
8. 计算每日指标。
9. 从第 3 天开始输出趋势报告。

推荐目录结构：

```text
data/
  raw/
    2026-06-22/
      keyword_露营装备.xlsx
      keyword_露营新手.xlsx
  images/
    2026-06-22/
  processed/
    2026-06-22.parquet
reports/
  2026-06-22_小红书趋势报告.md
  2026-06-22_小红书趋势报告.xlsx
config/
  domain_keywords.yaml
```

## 5. 用大模型生成关键词

可以用大模型，但不建议完全依赖大模型。最佳方式是“人工种子词 + 大模型扩展 + 数据反哺更新”。

输入给大模型：

```text
你是小红书运营研究员。我要调研的领域是：露营装备。
请从以下维度生成关键词：
1. 核心领域词
2. 场景词
3. 人群词
4. 需求词
5. 产品词
6. 情绪词
7. 决策词
8. 竞品/品牌词
每类给 10 个，并说明为什么这些词能覆盖领域。
```

每天分析后，还应从高赞标题、OCR 文本和聚类主题中提取新词，进入候选关键词池。

关键词池分三层：

- A 类核心词：每天抓，数量 10 到 20 个。
- B 类扩展词：隔天抓，数量 30 到 50 个。
- C 类探索词：每周轮换，数量 50 到 100 个。

## 6. 每天爬多少条合适

建议先从低风险、小规模开始：

- 单关键词：50 到 100 条。
- 每日关键词：10 到 30 个。
- 每日总量：500 到 2000 条搜索卡片。

第一周建议：

- 关键词数：10 个。
- 每个关键词：50 条。
- 每天总量：约 500 条。

稳定后再提高：

- 关键词数：20 个。
- 每个关键词：80 条。
- 每天总量：约 1600 条。

不建议一开始每天抓几万条。小红书搜索结果重复率高，且搜索页排序会偏向高互动内容，盲目加量不一定显著提高分析质量。

## 7. 是否指定爬取时间范围

搜索页本身不一定提供稳定的时间筛选，因此第一阶段不强依赖平台时间范围。

分析时使用两种时间：

- `crawl_date`：脚本运行日期，表示当天看到的搜索结果。
- `publish_date`：笔记发布时间，表示内容发布日期。

推荐筛选逻辑：

- 趋势分析：优先看最近 7 天、14 天、30 天发布的笔记。
- 热点分析：可以看当天抓到的所有高赞内容。
- 上升趋势：依赖连续多天的 `crawl_date` 数据。

如果 `publish_date` 缺失，不直接丢弃，可以进入“未知发布时间”分组，但不参与严格时间趋势计算。

## 8. 降重方案

降重要分两层：硬降重和语义降重。

### 8.1 硬降重

优先级：

1. `note_id` 完全相同：同一笔记，保留一条。
2. `link` 完全相同：同一笔记，保留一条。
3. `author + title` 完全相同：高度疑似重复，保留一条。
4. `cover_url` 完全相同且标题相似：疑似重复，进入人工或语义判断。

保留规则：

- 保留点赞数更高的一条。
- 如果同一笔记多天出现，保留每日出现记录，同时维护一张 `note_daily_snapshot` 表，用来分析排名和点赞变化。

### 8.2 语义降重

适用于标题不同但内容高度相似的笔记，例如：

- “露营新手必买清单”
- “新手露营装备清单”
- “第一次露营别乱买，这些够了”

语义降重方法：

1. 合并 `title + visible_text + OCR 文本 + 图片描述`。
2. 用 embedding 模型生成向量。
3. 计算余弦相似度。
4. 相似度高于 0.88 的内容归为同一主题簇。
5. 每个主题簇保留代表帖，同时保留簇内数量和互动总量。

注意：语义降重不是删除数据，而是把重复表达聚合成主题。

## 9. 图片解析方案

封面图对小红书趋势分析非常重要。很多爆点写在图上，而不在标题里。

推荐两级解析：

### 9.1 OCR

提取封面上的文字：

- 大标题
- 卖点词
- 数字词
- 标签词
- 品牌词
- 情绪词

例如：

```text
新手露营别乱买
这 8 件真的够了
平价但高级
```

### 9.2 多模态大模型

让模型对封面图输出结构化信息：

- 图片主体
- 场景
- 是否有人物
- 是否有产品展示
- 色调
- 构图类型
- 是否有大字标题
- 视觉风格
- 可能对应的内容类型
- 适合的人群

输出 JSON 示例：

```json
{
  "scene": "户外草地露营",
  "main_objects": ["帐篷", "折叠椅", "露营车"],
  "has_person": true,
  "visual_style": "清新生活方式",
  "cover_text": "新手露营装备清单",
  "content_type": "清单攻略",
  "target_audience": ["新手", "女生", "周末出游人群"]
}
```

## 10. 指标计算

分析必须有数值支撑，不能只让大模型写主观总结。

建议每天计算：

- 样本量：抓取笔记数。
- 有效样本量：去重后笔记数。
- 新笔记数：当天首次出现的 `note_id` 数。
- 近 7 天发布笔记占比。
- 平均点赞数。
- 中位数点赞数。
- P90 点赞数。
- 高赞笔记数：点赞数大于阈值，例如大于 1000。
- 高赞率：高赞笔记数 / 有效样本量。
- 主题簇数量。
- 每个主题簇的笔记数、点赞总量、平均点赞、最高点赞。
- 关键词覆盖率：每个关键词贡献的有效样本数。
- 作者集中度：Top 作者占比。
- 封面文字出现率。
- 高频词 Top 50。

趋势指标：

- 主题热度 = 主题笔记数权重 + 点赞总量权重 + 高赞率权重 + 近 7 天发布占比权重。
- 主题增速 = 今天主题热度 / 最近 3 天平均热度 - 1。
- 新兴主题 = 过去 2 天少见，今天样本数和点赞显著增加。
- 稳定热点 = 连续 3 天进入 Top 10 的主题。

示例公式：

```text
topic_score =
  log(1 + note_count) * 0.30
  + log(1 + total_likes) * 0.35
  + high_like_rate * 0.20
  + recent_publish_ratio * 0.15
```

## 11. 三天后开始分析

第一天和第二天只能做横截面分析，不能严肃判断趋势。

从第三天开始可以输出：

- 连续 3 天都热的稳定主题。
- 第 3 天突然上升的新主题。
- 近 3 天高赞标题模式。
- 近 3 天封面风格变化。
- 哪些关键词贡献了最多新主题。

从第七天开始可以输出：

- 周趋势。
- 主题生命周期。
- 高赞内容稳定结构。
- 上升和下降主题。
- 关键词池优化建议。

## 12. 大模型分析方案

大模型适合做：

- 主题命名
- 标题模式归纳
- 封面风格总结
- 痛点/需求提取
- 案例拆解
- 选题建议
- 关键词扩展

大模型不应直接负责：

- 原始去重
- 点赞数统计
- 日期计算
- 排名计算
- 趋势增速计算

正确方式是：先用程序算指标，再把结构化指标和代表样本交给大模型解释。

大模型输入应包含：

- 主题簇名称
- 簇内样本数
- 点赞总量
- 平均点赞
- 最高点赞
- 近 3 天变化
- 代表标题
- 代表封面 OCR
- 代表图片描述
- 代表链接

这样报告才有逻辑和证据。

## 13. 突出帖子案例分析

每个报告建议选 5 到 10 个案例。

选择规则：

- 点赞数 Top。
- 所属主题增速高。
- 标题或封面具有明显可复用结构。
- 与领域核心业务相关。
- 非明显广告或搬运内容。

案例分析内容：

- 标题
- 作者
- 发布时间
- 点赞数
- 链接
- 所属主题
- 标题结构
- 封面风格
- OCR 文字
- 目标人群
- 内容钩子
- 用户痛点
- 可复用选题角度
- 风险或不可复用因素

## 14. 是否进入详情页

不建议默认进入详情页。

原因：

- 小红书详情页更容易触发登录校验、验证码或滑块。
- 详情页加载更多接口和动态资源，行为更像采集。
- 少量点击也不能保证不触发风控，只能降低概率。
- 风控策略会变化，需要持续维护。

如果确实需要案例深挖，可以设计为“少量、人工确认、可关闭”的可选功能。

建议规则：

- 默认不开启详情页。
- 每日报告最多进入 3 到 5 个详情页。
- 只进入高价值案例。
- 两次详情页访问间隔 30 到 90 秒。
- 命中验证码立即停止。
- 失败不重试。
- 不滚动评论区。
- 不抓评论列表。
- 不做批量详情页补字段。

命令示例：

```powershell
conda run -n pt python analyze_xhs_trends.py --domain "露营装备" --detail-cases 3
```

更推荐的替代方案：

- 报告先列出案例链接。
- 用户手动打开少量案例查看。
- 工具只基于搜索页和封面图做分析。

## 15. 风控边界

本方案不做绕过平台风控的功能。

可做的低风险控制：

- 使用真实 Chrome 持久化登录态。
- 每天固定低频运行。
- 控制关键词数量和每词抓取数量。
- 慢速滚动。
- 不并发打开多个页面。
- 不默认进入详情页。
- 命中验证码或异常提示后停止。
- 不使用账号池、代理池、批量切号。

不能保证的事情：

- 不能保证永远不触发风控。
- 不能保证页面结构长期不变。
- 不能保证收藏数、评论数能从搜索页获得。
- 不能保证详情页少量点击完全安全。

## 17. 系统模块设计

建议拆成以下模块：

```text
collector/
  xhs_search_exporter.py       # 当前搜索页采集脚本

pipeline/
  run_daily.py                 # 每日调度入口
  keyword_planner.py           # 关键词生成和轮换
  image_downloader.py          # 封面下载
  image_analyzer.py            # OCR / 多模态解析
  cleaner.py                   # 清洗、标准化、硬降重
  semantic_deduper.py          # embedding 语义聚类
  metrics.py                   # 指标计算
  llm_analyzer.py              # 大模型分析
  report_writer.py             # Markdown / Excel 报告输出

config/
  domain_keywords.yaml
  analysis_config.yaml
```

## 19. MVP 开发顺序

第一版：

1. 每日多关键词采集。
2. 合并所有 Excel。
3. 硬降重。
4. 基础指标计算。
5. 高频词和高赞标题分析。
6. 输出 Excel 和 Markdown 报告。

第二版：

1. 下载封面图。
2. OCR 提取封面文字。
3. 多模态解析封面。
4. 大模型主题聚类和命名。
5. 案例拆解。

第三版：

1. 连续 3 天趋势分析。
2. 关键词池自动扩展。
3. 主题增速计算。
4. 每周报告。

第四版：

1. 可选少量详情页案例深挖。
2. 人工确认后打开。
3. 命中验证码立即停止。

## 21. 结论

仅依靠搜索结果页数据，可以做出有实际运营价值的小红书趋势分析功能。

第一阶段不要追求详情页字段，而应把重点放在：

- 连续每日采集
- 关键词覆盖
- 封面 OCR 和多模态解析
- 降重和主题聚类
- 数值化趋势指标
- 有证据的案例拆解

详情页分析可以作为后续可选能力，但不应作为基础流程的一部分。少量点击也不能保证完全不触发风控，因此默认关闭更稳。

## 22. 参考 GitHub Skill / MCP 项目的优化思路

调研 GitHub 上的小红书 Skill / MCP 项目后，可以提炼出几类值得借鉴的设计逻辑。

参考项目包括：

- `DeliciousBuding/xiaohongshu-skill`：Python + Playwright，强调从页面状态中抽结构化数据，输出纯 JSON，并兼容 AgentSkills。
- `chenningling/Redbook-Search-Comment-MCP2.0`：基于 Playwright 的 MCP Server，将笔记分析、AI 生成和发布动作拆成模块。
- `autoclaw-cc/xiaohongshu-mcp-skills`：把能力拆成多个 skill，包括登录、搜索、浏览、互动、主页、内容策划。
- `Xiangyu-CAS/xiaohongshu-ops-skill`：偏运营助手，强调推荐流分析、账号分析、选题灵感和知识库沉淀。

这些项目的共同思路不是“单个脚本抓完所有东西”，而是把能力拆成可组合模块：登录态管理、搜索采集、内容抽取、分析、报告、知识库、可选动作。

## 23. 对当前方案最值得借鉴的点

### 23.1 输出结构化 JSON，而不只依赖 Excel

Excel 适合人工查看，但不适合长期自动化分析。建议每日采集后同时保存：

- 原始 Excel：给人看。
- 原始 JSONL：给程序复跑和调试。
- 处理后 Parquet / CSV：给指标计算。
- Markdown 报告：给运营复盘。

推荐每条笔记保存为稳定 schema：

```json
{
  "crawl_date": "2026-06-22",
  "keyword": "露营装备",
  "rank": 12,
  "note_id": "...",
  "link": "...",
  "title": "...",
  "author": "...",
  "publish_time": "3天前",
  "publish_date": "2026-06-19",
  "like_count": 1288,
  "cover_url": "...",
  "visible_text": "...",
  "extract_method": "search_card_dom",
  "quality_flags": []
}
```

### 23.2 记录抽取方法和质量标记

GitHub 上不少项目会做多种内容获取方法和错误处理。我们不一定要进入详情页，但可以借鉴“多策略抽取 + 质量标记”。

搜索页字段建议记录：

- `extract_method`：字段来自 DOM、可见文本正则、图片 OCR、模型解析还是历史补全。
- `quality_flags`：例如 `missing_title`、`missing_publish_date`、`missing_like_count`、`duplicate_candidate`、`ocr_failed`。
- `raw_snapshot_path`：必要时保存少量 HTML 片段或截图，方便页面结构变化时修复。

这样后续分析时可以知道哪些数据可信，哪些只是兜底推断。

### 23.3 MCP / Skill 式模块拆分

当前项目可以不马上实现 MCP Server，但架构上可以按 MCP 工具思路拆函数。未来接入 Codex、Claude、OpenClaw 或其他 Agent 时会更轻松。

建议抽象成这些“工具能力”：

```text
xhs.login_status
xhs.search_notes
xhs.download_covers
xhs.extract_image_text
xhs.cluster_topics
xhs.compute_metrics
xhs.generate_report
xhs.suggest_keywords
xhs.update_knowledge_base
```

对应到本地脚本：

```text
python -m pipeline.search --domain 露营装备
python -m pipeline.images --date 2026-06-22
python -m pipeline.analyze --date 2026-06-22
python -m pipeline.report --date 2026-06-22
```

### 23.4 两阶段分析，而不是一步生成报告

不少 MCP 项目会先获取笔记分析结果，再交给 AI 做生成或下一步动作。趋势分析也应采用两阶段：

第一阶段，程序计算：

- 去重结果
- 主题簇
- 点赞统计
- 高频词
- 增速
- 异常值
- 代表样本

第二阶段，大模型解释：

- 为什么这个主题可能升温
- 哪些标题结构可复用
- 哪些封面风格值得关注
- 运营上下一步怎么做

大模型不直接决定趋势，而是解释趋势。

### 23.5 知识库沉淀

运营类 skill 很强调知识库，这是非常值得借鉴的。趋势分析不是一次性报告，而是每天积累判断。

建议维护一个 `knowledge-base/`：

```text
knowledge-base/
  domain_profile.md              # 领域定义、人群、产品范围
  keyword_pool.md                # 关键词池和权重
  topic_taxonomy.md              # 已知主题树
  title_patterns.md              # 高赞标题模板
  cover_patterns.md              # 封面风格模板
  weekly_insights/
    2026-W26.md
```

每次报告结束后，把稳定结论写回知识库：

- 新增主题
- 过时主题
- 高赞标题模板
- 封面风格变化
- 关键词增减建议

## 24. 我不建议照搬的点

### 24.1 不建议做互动能力

很多小红书 MCP / skill 项目包含点赞、收藏、评论、发布等功能。你的目标是趋势分析，不是账号自动运营。互动能力会显著增加风控风险，也会让项目边界变复杂。

当前项目建议明确不做：

- 自动点赞
- 自动收藏
- 自动评论
- 自动关注
- 自动发布
- 批量私信

### 24.2 不建议默认采详情页

有些项目为了拿正文、评论和完整互动字段，会进入详情页或调用详情接口。对趋势分析来说，这不是第一优先级。

我的建议：

- 搜索页数据用于规模化趋势判断。
- 封面 OCR / 多模态用于补充内容理解。
- 详情页只作为人工确认后的少量案例研究。
- 不把详情页字段放进核心指标体系。

### 24.3 不建议依赖反检测作为核心能力

有项目会加入 stealth、指纹、变速滚动等能力。我们可以借鉴“慢速、自然、持久化登录态、异常停止”这些温和策略，但不应把绕风控能力作为产品核心。

原因：

- 维护成本高。
- 方案实时变化。
- 账号风险不可控。
- 和趋势分析目标不匹配。

## 28. 优化后的报告逻辑

报告应该像运营分析，而不是爬虫结果摘要。

每个结论必须有三类证据：

1. 数据证据：样本数、点赞数、增速、占比。
2. 内容证据：代表标题、封面 OCR、主题簇关键词。
3. 案例证据：代表链接和为什么它突出。

示例：

```text
结论：新手露营“少买但买对”正在升温。

数据证据：
- 近 3 天该主题样本数从 18 增至 42，增长 133%。
- 平均点赞 812，高于全域平均 356。
- 高赞率 19%，高于全域 7%。

内容证据：
- 高频词：新手、别乱买、够了、避坑、平替。
- 封面 OCR 高频句式：第一次露营 / 这几件够了 / 千万别买。

代表案例：
- 《第一次露营别乱买，这 8 件真的够了》，点赞 3200。
- 突出原因：标题有明确人群 + 反常识建议 + 数字清单，封面强化“省钱避坑”。
```

## 29. 我的建议

我建议你不要把这个项目做成“小红书万能自动化工具”，而要做成“小红书趋势研究流水线”。

理由：

- 越接近互动、发布、详情页、评论区，风控越高。
- 趋势分析真正需要的是连续、稳定、可对比的数据，不是一次抓到最全字段。
- 搜索页卡片 + 发布时间 + 点赞数 + 封面信息，已经足够做运营级趋势判断。
- 大模型最适合做解释和归纳，不适合替代数据计算。
- 知识库沉淀会让系统越用越懂你的领域，这是比多抓几个详情字段更有价值的资产。

最终建议路线：

1. 第一周：只做搜索页每日采集、硬降重、基础指标、日报。
2. 第二周：加入 OCR、多模态封面解析、语义聚类。
3. 第三周：加入关键词池自动扩展和知识库。
4. 第四周：只在人工确认时，少量打开详情页做案例校验。

这个路线更稳，也更贴合你的真实目标：不是“爬更多”，而是“更早发现值得做的内容机会”。
