## 1. 项目定位

本项目要做的是“小红书搜索页市场局势追踪工具”，不是自由行动的 ReAct agent。

核心流程：

```text
搜索页采集 -> 清洗去重 -> 指标/规则信号 -> 证据索引 -> 记忆更新 -> LLM 解释 -> 报告
```

核心边界：

- 主流程只依赖搜索页数据。
- 详情页、OCR、封面图分析都是可选增强。
- LLM 只解释结构化数据，不直接分析完整原始表。
- 报告分析现有局势，建议内容低优先级。

## 2. 现有代码基线

当前已有能力：

- `pipeline/run_daily.py`：按领域采集搜索页并保存 `all_raw`。
- `pipeline/clean_notes.py`：清洗、硬去重、基础质量分、简单 topic。
- `pipeline/compute_metrics.py`：基础指标、top notes、topic metrics。
- `pipeline/generate_report.py`：生成现有日报。
- `xhs_trend_app.py`：PySide6 GUI 串联现有流程。

可以继续改，不需要把现有代码当成不可动的最终架构。但改造要保持现有流程阶段性可运行。

## 3. 必须保留的设计原则

- Pipeline-first：业务流程由 Python 显式编排。
- 多领域隔离：所有领域数据写入 `projects/<domain>/`。
- 事实层由程序写入：LLM 不直接改 CSV/JSON 指标表。
- 证据可追溯：重要结论必须引用 `evidence_id`。
- 数据质量降级：低质量样本不能输出强趋势判断。
- 搜索页边界：不得把搜索页样本写成全市场、销量、成交、投放事实。
- 记忆压缩：长期追踪依赖 daily summary、rollup、current_state，不把全部历史塞给 LLM。

## 4. 核心语义

### topic

`topic` 表示讨论对象或需求主题。

例子：

- 某 IP 周边
- AI 写作
- AI 绘画
- 通勤包
- 桌搭收纳

### content_pattern

`content_pattern` 表示内容打法或标题类型。

例子：

- 测评
- 避坑
- 清单
- 教程
- 对比
- 平替
- 开箱

禁止把“测评/避坑/清单”写入 `topic_daily.csv`。内容打法应写入 pattern 相关结果。

## 7. 关键数据契约

### note_global_id

同一篇笔记跨日、跨关键词的稳定身份。

生成优先级：

```text
note_id -> link -> title + author
```

### evidence_id

某一天、某领域、某次搜索样本中的证据记录。

格式：

```text
ev_<yyyymmdd>_<domain>_<source>_<hash>
```

规则：

- 同一天同一领域内，同一 `note_global_id` 只生成一个 `evidence_id`。
- 跨日重复出现时生成新的 `evidence_id`，但关联同一个 `note_global_id`。
- 同一笔记来自多个关键词时，在 evidence 中记录多个 keywords。

### data_quality

质量等级：

```text
high / medium / low / invalid
```

默认降级规则：

- `clean_count < 3`：invalid，只生成数据质量报告。
- `clean_count < 10`：low，允许弱报告，不更新长期趋势判断。
- 发布时间缺失率过高：禁用 rising/cooling 强判断。
- 互动字段缺失率过高：互动相关排序和结论降级。

## 9. LLM 使用方式

LLM 输入必须由程序压缩生成，例如：

```text
projects/<domain>/processed/YYYY-MM-DD_llm_input.json
```

输入包含：

- 数据边界
- 数据质量
- current_state 摘要
- 显著变化
- topic/author/pattern/entity signals
- representative evidence

LLM 输出必须可校验：

- JSON 优先。
- 重要判断带 `evidence_id`。
- 不确定内容进入 uncertainties。
- 禁止输出销量、成交、投放、全市场结论。

## 10. 记忆系统

记忆分层：

- raw/processed：事实源，不给 LLM 全量读取。
- daily summary：每日短摘要。
- rollup：周/月结构化统计和可读摘要。
- current_state：当前领域压缩认知，覆盖式更新。
- evidence：证据索引。
- judgments：历史判断、冲突处理和状态变更。

冲突处理：

- 由程序基于结构化指标触发。
- 低质量日只进入待观察，不覆盖长期判断。
- 连续多天确认后再更新 current_state。

## 12. 测试重点

通用测试写法不用在方案里展开，但以下项目特有逻辑必须测：

- `note_global_id` 和 `evidence_id` 稳定性。
- 同日跨关键词去重。
- 跨日 evidence 关联。
- topic 与 content_pattern 不混写。
- data_quality 降级。
- LLM 输入压缩不会包含完整 raw/clean 表。
- 多领域隔离。
- 低质量数据不覆盖 current_state。

测试命令遵守 `AGENTS.md`：

```powershell
conda run -n pt python -m unittest discover -s tests
```

## 14. 成功标准

- 不依赖详情页，也能稳定输出搜索页市场局势分析。
- 一个领域连续运行多天后，可以看到 topic、作者、content_pattern 的变化。
- 每条重要判断可以追溯 evidence。
- 报告不会夸大搜索页样本。
- 多领域记忆互不污染。
- 长期运行后 LLM 上下文仍可控。
- 低质量数据会自动降级。

## 15. 核心一句话

让 Python 管事实、证据、规则、记忆和流程；让 LLM 只在证据边界内解释局势。
