# 小红书趋势分析工具开发方案（代码改造版）

本文档用于指导后续 AI 继续开发当前代码库。它不假设现有代码完美，也不要求完全推翻重写；原则是保留能工作的采集和基础指标能力，在其上逐步改造成“搜索页市场局势追踪工具”。

## 1. 产品定位

目标不是做一个能自由行动的 ReAct agent，而是做一个 pipeline-first 的市场局势分析系统：

```text
搜索页采集 -> 清洗去重 -> 规则/指标计算 -> 证据索引 -> 记忆更新 -> LLM 受约束解释 -> 报告
```

LLM 的角色：

- 解释结构化事实。
- 生成可读报告。
- 辅助压缩 current_state/wiki。
- 不直接爬取、不直接改事实表、不自由创造结论。

系统边界：

- 核心能力建立在小红书搜索页，不依赖详情页。
- 详情页、封面图、OCR 都是可选增强，失败不影响主流程。
- 报告只写“搜索页样本显示”，不写全市场销量、成交、投放等不可见事实。

## 2. 当前代码现状

现有可保留能力：

- `collector/search_page_collector.py`：搜索页采集，支持标题、作者、链接、note_id、互动字段等。
- `pipeline/run_daily.py`：按领域配置批量采集，并生成当天 `all_raw.xlsx/json`。
- `pipeline/clean_notes.py`：字段标准化、硬去重、质量分、简单 topic。
- `pipeline/compute_metrics.py`：基础指标、top notes、topic metrics、候选关键词。
- `pipeline/generate_market_report.py`：读取结构化输入包，生成市场局势主报告。
- `xhs_trend_app.py`：PySide6 GUI，可配置领域、扩展关键词、串联运行流程。
- `tests/test_core.py`：已有基础测试，可继续扩展。

主要不足：

- GUI 只是串行跑命令，没有 pipeline 状态机和结构化错误记录。
- 历史 clean 合并已统一为 `merge_history_clean.py`。
- `topic_cluster.py` 只是用高频词做粗糙匹配，没有领域规则、优先级、冲突处理。
- 没有统一 `evidence_id`、`note_global_id`、证据索引和跨日去重。
- 没有数据质量降级文件，低质量数据仍可能进入强判断。
- 没有长期记忆层：daily summary、rollup、current_state、judgments 都未实现。
- LLM 报告仍偏“拿 metrics 直接总结”，缺少 schema 校验和 evidence 校验。
- 多领域已按目录隔离，但规则/taxonomy 的全局与领域覆盖关系未定义。

## 3. 改造总原则

1. 保留现有可运行链路，每次只替换一个可验证环节。
2. 新增模块优先，避免大规模改写旧模块导致现有 GUI 不能运行。
3. 事实层只能由 Python 写入，LLM 只产生解释文本或候选摘要。
4. 所有重要结论都要有 `evidence_id`。
5. 所有新增模块都要支持 `--domain` 和 `--date`。
6. 每个 pipeline 步骤都要有输入、输出、状态、错误摘要。
7. 规则默认可复现，模型分类只能作为可选增强。

## 5. Pipeline 目标顺序

推荐目标顺序：

```text
pipeline.run_daily
pipeline.clean_notes
pipeline.analyze_images --optional
pipeline.compute_metrics
pipeline.compute_search_page_signals
pipeline.generate_evidence
pipeline.evaluate_data_quality
pipeline.merge_history_clean
pipeline.update_memory
pipeline.generate_market_report
```

说明：

- `run_daily` 已经会生成当天 `all_raw`，不需要另一个 `merge_raw` 合并当天原始数据。
- `merge_history_clean.py` 的职责是生成 `history_clean_notes.csv`。
- `analyze_images` 只作为可选增强，不应默认阻塞。
- `compute_search_page_signals` 负责内容打法、作者占位、实体候选、需求信号。
- `generate_evidence` 统一生成证据 ID 和证据文件。
- `evaluate_data_quality` 决定报告和记忆更新的强弱。
- `update_memory` 只读取结构化结果，不重新分析原始表。
- `generate_market_report` 只读取 LLM 输入包和结构化证据，不直接读取全部原始数据。

## 6. 新增和改造模块

### 6.1 Pipeline 状态机

新增：

```text
pipeline/runner.py
pipeline/status.py
```

输出：

```text
projects/<domain>/processed/YYYY-MM-DD_pipeline_status.json
```

字段：

```json
{
  "run_id": "run_20260624_143022",
  "domain_id": "ai_peripheral",
  "date": "2026-06-24",
  "status": "partial_success",
  "steps": [
    {
      "step": "run_daily",
      "status": "success",
      "started_at": "2026-06-24T14:30:22",
      "ended_at": "2026-06-24T14:35:10",
      "input": {"keywords": 10},
      "output": {"raw_files": 10, "total_notes": 487},
      "errors": []
    }
  ]
}
```

第一版不做复杂“一键从失败步骤恢复”，但要把失败步骤、错误摘要、已生成文件记录清楚。

### 6.2 规则系统

新增：

```text
analysis/rule_loader.py
analysis/rule_matcher.py
memory_global/pattern_rules/*.yaml
```

规则 YAML 示例：

```yaml
version: 1
rules:
  - id: content_review_v1
    label: 测评
    type: content_pattern
    keywords: ["测评", "实测", "真实体验"]
    priority: 80
    confidence: 0.8
    effective_from: "2026-01-01"
    expires_at: null
```

规则输出必须包含：

- `label`
- `rule_id`
- `confidence`
- `matched_text`
- `priority`
- `ambiguous`

冲突处理：

- 同类标签按 priority 取主标签。
- 多个合理标签可保留在数组中。
- 模糊样本标记 `ambiguous: true`，不进入强判断。

### 6.3 Topic 与 Content Pattern

改造：

```text
analysis/topic_cluster.py
analysis/content_pattern_analyzer.py
```

边界：

- `topic`：讨论对象或需求主题，例如“AI 写作”“某 IP 周边”“通勤包”。
- `content_pattern`：内容打法，例如“测评”“避坑”“清单”“教程”。
- `hybrid_tags`：topic 与 pattern 的组合，例如“某 IP 周边 + 测评”。

输出字段：

```text
topic_cluster_id
topic_name
topic_confidence
topic_matched_rules
content_patterns
primary_content_pattern
pattern_confidence
hybrid_tags
ambiguous_tags
```

### 6.4 Evidence 系统

新增：

```text
analysis/evidence.py
pipeline/generate_evidence.py
```

身份层级：

- `note_global_id`：同一篇笔记跨天、跨关键词的稳定身份。
- `evidence_id`：某一天、某领域、某次搜索结果中的证据记录。

规则：

- 同一天同一领域内，同一 `note_global_id` 只生成一个 `evidence_id`。
- 如果同一笔记来自多个关键词，`keywords` 保存数组。
- 跨日重复出现，生成新的 `evidence_id`，但关联同一个 `note_global_id`。

`evidence_id` 格式：

```text
ev_<date>_<domain>_<source>_<short_hash>
```

`note_global_id` 格式：

```text
note_<short_hash>
```

`YYYY-MM-DD_evidence.jsonl` 字段：

```json
{
  "evidence_id": "ev_20260624_ai_search_a1b2c3d4",
  "note_global_id": "note_9f8e7d6c",
  "date": "2026-06-24",
  "domain_id": "ai",
  "source": "search_page",
  "keywords": ["AI 写作", "AI 办公"],
  "best_rank": 2,
  "all_ranks": [{"keyword": "AI 写作", "rank": 2}],
  "note_id": "",
  "link": "",
  "title": "",
  "author": "",
  "metrics": {},
  "topic": {},
  "content_patterns": [],
  "entity_candidates": [],
  "quality_flags": []
}
```

### 6.5 数据质量

新增：

```text
analysis/data_quality.py
pipeline/evaluate_data_quality.py
```

质量等级：

```text
high / medium / low / invalid
```

默认规则：

- `clean_count < 3`：`invalid`，不生成市场局势报告。
- `clean_count < 10`：`low`，可生成弱报告，不更新长期趋势判断。
- `missing_like_rate > 0.5`：互动指标降级。
- `publish_date_present_rate < 0.3`：近期性和升降温判断降级。
- `dedupe_rate > 0.7`：重复严重，前排占位判断降级。
- 连续采集失败：触发当天熔断。

### 6.6 搜索页信号

新增：

```text
analysis/search_page_signals.py
analysis/author_intelligence.py
analysis/entity_miner.py
pipeline/compute_search_page_signals.py
```

输出：

```text
projects/<domain>/processed/YYYY-MM-DD_search_signals.json
```

信号类型：

- topic signals
- author occupation signals
- content pattern signals
- entity/IP/product candidate signals
- demand signals
- anomaly signals

每个信号必须包含：

- `signal_id`
- `signal_type`
- `target`
- `sample_count`
- `signal_strength`
- `confidence`
- `evidence_ids`
- `data_quality_level`

`signal_strength` 建议由程序计算：

```text
样本量得分 + 持续天数得分 + 变化幅度得分 + 前排权重 + 数据质量权重
```

### 6.7 记忆系统

新增：

```text
pipeline/update_memory.py
analysis/memory_store.py
analysis/memory_context.py
analysis/conflict_resolver.py
```

写入内容：

- daily summary
- topic/keyword/author/pattern daily CSV
- trend_events.jsonl
- judgments.jsonl
- conflict_resolution_queue.jsonl
- weekly/monthly metrics
- current_state.md
- wiki index/log

冲突规则：

- 冲突检测由程序触发，基于结构化指标。
- 低质量日只进入 `conflict_resolution_queue`，不覆盖 current_state。
- 连续 3 天 cooling 才覆盖 rising。
- `judgments.jsonl` 增加 `resolution_status`：
  - `pending`
  - `resolved`
  - `overridden`
  - `ignored_low_quality`

冷启动：

- 新领域首次运行，生成 `status: initial` 的 current_state。
- 前 3 天趋势判断都标记 `needs_verification`。

重新激活：

- 领域 30 天以上未运行，状态为 `reactivated`。
- 读取最近 rollup，但新判断至少观察 3 天。

### 6.8 LLM 输入压缩

新增：

```text
analysis/llm_context_builder.py
```

不要把全部 metrics 和 evidence 塞给 LLM。输入包应由程序生成：

```text
projects/<domain>/processed/YYYY-MM-DD_llm_input.json
```

选择 evidence 的原则：

- 覆盖不同 topic。
- 覆盖不同 content_pattern。
- 包含高互动样本。
- 包含高频但不高互动样本。
- 包含新出现样本。
- 包含异常样本。

LLM 输入建议：

```json
{
  "data_boundary": "",
  "data_quality": {},
  "current_state": "",
  "significant_changes": [],
  "topic_landscape": [],
  "author_landscape": [],
  "content_pattern_landscape": [],
  "entity_landscape": [],
  "evidence_summary": [],
  "representative_evidence": []
}
```

### 6.9 市场局势报告

新增或替换：

```text
pipeline/generate_market_report.py
```

报告结构：

```text
执行摘要
数据边界和数据质量
主题格局
前排占位
内容打法
需求信号
实体/IP/产品候选
关键变化
不确定性
证据样本
低优先级行动提示
```

要求：

- 执行摘要 200 字以内。
- 行动建议只放最后，且占比低。
- 每条重要结论引用 evidence_id。
- 如果 data_quality 为 low，只生成弱判断。
- 如果 invalid，只生成数据质量报告。

## 8. 调度

第一阶段以手动触发为主。

可选支持 Windows 任务计划程序：

- 每个 domain 配置 `schedule_enabled`。
- 配置 `preferred_time`。
- 同一 domain 默认每天只完整运行一次。
- 上次熔断未解除时自动跳过。

Rollup 触发：

- Weekly：该周有效数据不少于 3 天时生成。
- Monthly：该月有效数据不少于 15 天时生成。
- metrics JSON 由程序生成。
- summary Markdown 可由 LLM 生成，但输入只允许使用 rollup metrics JSON。

## 11. AI 开发注意事项

后续 AI 开发必须避免：

- 直接把完整 DataFrame 丢给 LLM。
- 让 LLM 自己决定 topic、账号类型、趋势状态并写入事实表。
- 把 content_pattern 写进 topic_daily。
- 让详情页采集成为主流程依赖。
- 一次性重构 GUI 和 pipeline。
- 无 schema 地写 JSON/CSV。
- 忽略 Windows 中文编码问题。
- 删除或覆盖用户已有数据。

每次代码改动后至少做：

```powershell
conda run -n pt python -m py_compile <changed_file.py>
conda run -n pt python -m unittest discover -s tests
```

## 12. 成功标准

第一阶段成功：

- 现有采集、清洗、指标、市场局势报告仍能跑通。
- 每次运行有 pipeline 状态文件。
- 每条代表性结论有 evidence_id。
- 数据质量差时报告会降级。

第二阶段成功：

- 一个领域连续运行多天后，可以看到 topic、作者、content_pattern 的变化。
- current_state 不无限增长。
- 多领域记忆互不污染。
- LLM 输入保持短小、结构化、可追溯。

最终成功：

- 不依赖详情页，也能稳定输出“当前搜索页局势”。
- 报告能区分事实、信号、不确定性。
- 规则可维护、可评估、可被人工纠正。
- 长期运行后仍然可解释、可测试、可迭代。

## 13. 核心一句话

本项目下一阶段不是把 LLM 权限放大，而是把数据事实层做扎实：

> 让 Python 管事实、证据、规则、记忆和流程，让 LLM 只在证据边界内解释局势；在保留现有可运行代码的基础上，逐步升级成可长期追踪、可低幻觉分析的小红书搜索页市场局势系统。
